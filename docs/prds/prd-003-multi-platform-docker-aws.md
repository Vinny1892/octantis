## 1. Context

- **Product / System**: Octantis — an intelligent infrastructure monitoring agent.

- **Current State**: Octantis (as of PRD 002) receives OTLP telemetry events, filters them through a 2-stage pipeline (TriggerFilter + FingerprintCooldown), and runs a LangGraph investigation workflow where the LLM queries Prometheus and Loki via Grafana MCP. The entire stack assumes a Kubernetes environment — TriggerFilter rules target K8s-specific signals (CrashLoopBackOff, OOMKill, pod evictions), and the only investigation MCP is Grafana. The project is developed by a single person and is not yet in production.

- **Problem**: Octantis is only useful to operators running Kubernetes. Teams deploying workloads on Docker or AWS EC2/ECS cannot benefit from Octantis's intelligent investigation — even if they already have Node Exporter + OTel Collector sending metrics via OTLP, Octantis has no tools to investigate at the Docker or AWS infrastructure level. The TriggerFilter also lacks rules for host-level metrics (`node_cpu_seconds_total`, `node_memory_MemAvailable_bytes`, etc.), so Node Exporter events are silently dropped. This restricts Octantis to K8s-native environments and excludes the majority of smaller and mid-size teams who run Docker Compose or VM-based deployments.

## 3. Proposed Solution

### Overview

Expand Octantis to support Docker and AWS environments alongside existing Kubernetes support. An operator running Node Exporter on a Docker host or AWS EC2 instance will be able to send metrics via OTel Collector to Octantis and receive intelligent investigation reports and fix plans — the same experience K8s operators already have.

Three additions make this possible: (1) updating the TriggerFilter to recognize host-level metrics from Node Exporter; (2) adding environment detection so Octantis knows whether an event came from K8s, Docker, or AWS; and (3) adding Docker MCP and AWS MCP as investigation tools. MCPs are organized into two slots — **observability** (Grafana, future ELK) and **platform** (Docker, AWS, K8s). The free tier allows up to 1 observability + 1 platform MCP, with at least one MCP required. Combinations beyond 1+1 are reserved for the paid tier.

### Key Decisions

1. **Node Exporter ingestion via existing OTLP path** — Node Exporter data arrives via OTel Collector → OTLP → Octantis. No new ingestion protocol. Only TriggerFilter rules need updating to handle host-level metric names and thresholds.
2. **Environment detection from OTLP resource attributes + explicit config** — Detect platform using OTLP resource attributes (e.g., `cloud.provider=aws`, `container.runtime=docker`, `k8s.pod.name`) and support explicit override via `OCTANTIS_PLATFORM` env var for operators who cannot control OTel Collector config.
3. **Single active environment per Octantis instance** — One instance targets one platform (K8s, Docker, or AWS). This keeps routing logic simple and avoids concurrent MCP connections to unrelated platforms. Running multiple instances is the recommended pattern for mixed environments.
4. **Docker MCP and AWS MCP as greenfield integrations** — Neither is currently running or tested. Both require research to identify the right MCP server before implementation.
5. **LangGraph investigation workflow unchanged** — The workflow (investigate → analyze → plan → notify) is not modified. Only the set of MCP tools available to the LLM differs based on configuration.
6. **MCP slot model: 1 observability + 1 platform** — MCPs are categorized into two slots: **observability** (Grafana, future ELK) and **platform** (Docker, AWS, K8s, future VMware/GCP/Azure). The free tier allows at most 1 observability MCP + 1 platform MCP. At least one MCP (from either slot) must be configured — Octantis refuses to start with zero MCPs. Combinations beyond 1+1 (e.g., Grafana + ELK, or Docker + AWS simultaneously) are reserved for the paid tier. This enables future observability backends (e.g., ELK stack) without architectural changes.

### Out of Scope

- **Multiple MCPs per slot** — e.g., Grafana + ELK (2 observability), or Docker + AWS (2 platform) on the same instance. Planned for the paid tier.
- **Simultaneous multi-environment investigation** — An instance targeting Docker will not also query AWS MCPs for the same event. Planned for the paid tier.
- **Mixed environment on a single instance** — Handling Docker + AWS events on the same Octantis process simultaneously. Planned for the paid tier.
- **VMware, GCP, Azure** — Deferred to a future PRD. Docker and AWS are the initial targets due to testability. The pattern established here will apply to other platforms.
- **OTel Collector deployment** — Operators are assumed to already have OTel Collector running and forwarding Node Exporter data via OTLP.
- **Node Exporter deployment** — Assumed already installed on target hosts.
- **AWS workloads beyond EC2 and ECS** — Lambda, RDS, S3, and EKS (covered by the K8s path) are out of scope for v1.

## 8. Risks

| Risk | Impact | Mitigation |
|------|--------|------------|
| No official Docker MCP server exists or is difficult to integrate | High — M2 blocked | Research available Docker MCP servers before starting M2. Fallback: wrap Docker SDK as a custom MCP-compatible tool server. |
| AWS MCP server requires complex IAM setup | Medium — delays M3 | Start with minimal read-only IAM policy (EC2, CloudWatch, ECS describe/list). Document required permissions clearly in onboarding docs. |
| Node Exporter metric names vary by version or OS | Medium — triggers missed | Cover the most common metric names. Add configurable metric name patterns as env vars for operators to extend. |
| Auto-detection misidentifies EKS (has both K8s and AWS attributes) | Low | K8s detection takes priority in the detection order. Document this clearly. Explicit `OCTANTIS_PLATFORM` override always wins. |
| Adding multiple MCP connections increases startup time | Low | MCPManager already uses async connections (pattern from Grafana MCP). Health checks are non-blocking. |
| Operator tries to configure 2+ MCPs per slot on free tier | Low — startup blocked | Clear error message at startup explaining the slot model and pointing to paid tier for multi-MCP support. |

## 10. Architecture Overview

```
Node Exporter ──► OTel Collector ──OTLP──► ┌──────────────────────────────────────────────┐
(Docker host / EC2)                        │  Octantis                                    │
                                           │  OTLP Receiver (existing)                    │
                                           │       ↓                                      │
                                           │  TriggerFilter (updated: + node metrics)     │
                                           │       ↓                                      │
                                           │  FingerprintCooldown (existing)              │
                                           │       ↓                                      │
                                           │  Environment Detector (new)                  │
                                           │   OTLP attributes → k8s | docker | aws       │
                                           │   OCTANTIS_PLATFORM override                 │
                                           │       ↓                                      │
                                           │  LangGraph Workflow                          │
                                           │  ┌─────────────────────────────────────────┐ │
                                           │  │ investigate                             │ │
                                           │  │  ├─ Grafana MCP  (optional)        ◄───┼─┼──► Grafana
                                           │  │  ├─ Docker MCP   (optional)        ◄───┼─┼──► Docker
                                           │  │  ├─ AWS MCP      (optional)        ◄───┼─┼──► AWS
                                           │  │  └─ (future: ELK MCP, K8s MCP...) │    │ │
                                           │  ├─────────────────────────────────────────┤ │
                                           │  │ analyze → plan → notify                │ │
                                           │  └─────────────────────────────────────────┘ │
                                           └──────────────────────────────────────────────┘

MCP slot model: max 1 observability + 1 platform. At least 1 required.
Observability: Grafana, future ELK  |  Platform: Docker, AWS, K8s
Multiple MCPs per slot → paid tier.
```

- **TriggerFilter** (updated): Adds recognition for Node Exporter host-level metrics alongside existing K8s-specific signals.
- **Environment Detector** (new): Determines target platform from OTLP resource attributes or `OCTANTIS_PLATFORM` env var. Provides platform context to the LLM investigation prompt.
- **MCPManager** (updated): Manages MCP connections using a slot model. Enforces at least 1 MCP configured and max 1 per slot (observability / platform) in the free tier. The LLM receives tools from all configured MCPs.
- **Docker MCP** (new, platform slot): Container inspection, logs, resource stats.
- **AWS MCP** (new, platform slot): EC2 inspection, CloudWatch metrics, ECS task status.
- **Grafana MCP** (existing, observability slot): Prometheus (PromQL) and Loki (LogQL) queries. Platform-agnostic.
- **Future MCPs**: ELK (observability slot), VMware, GCP, Azure (platform slot) — same plug-in pattern, no architectural changes needed.
