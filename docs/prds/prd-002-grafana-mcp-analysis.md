## 1. Context

- **Product / System**: Octantis — an intelligent infrastructure monitoring agent for EKS/Kubernetes.

- **Current State**: Octantis receives OTLP telemetry events (metrics and logs) via gRPC/HTTP, passes them through a 3-layer filtering pipeline (PreFilter → EventBatcher → Sampler), enriches them with static Prometheus and Kubernetes API queries in the `collector` node, and then sends the enriched data to the LLM for severity classification. The LLM sees only the data that the code decided to fetch — a fixed set of PromQL queries and K8s API calls hardcoded in `collectors/prometheus.py` and `collectors/kubernetes.py`. The pipeline has 5 pre-filter rules, time-window batching (30s), and fingerprint-based deduplication (5-minute cooldown).

- **Problem**: The current architecture has two fundamental limitations:
  1. **The LLM analyzes stale, pre-selected data.** The `collector` node fetches a fixed set of metrics (CPU, memory, error rate, restarts, latency p99) before the LLM sees anything. If the real problem is disk pressure, network errors, or a specific application metric, the LLM has no way to ask for more data — it can only work with what the code pre-fetched. This creates blind spots where the LLM correctly identifies "something is wrong" but lacks the data to diagnose the root cause.
  2. **The pipeline is over-engineered for a trigger model.** The EventBatcher and Sampler were designed when every event was individually analyzed by the LLM. With the new model — events are triggers, not data — batching events by workload and merging metrics is unnecessary overhead. The LLM will query real-time data from the source when it needs to investigate.

  Moving to a trigger-based model where the LLM has direct access to observability data via Grafana MCP transforms Octantis from a "react to pre-digested data" agent into an "investigate autonomously" agent.

## 3. Proposed Solution

### Overview

Transform Octantis from a pipeline that pre-fetches data for the LLM into a system where OTLP events act as **triggers** and the LLM **autonomously investigates** by querying Prometheus (metrics) and Loki (logs) via Grafana MCP. The LLM decides what data to fetch, what queries to run, and how deep to investigate — it is no longer limited to a fixed set of pre-fetched metrics.

The filtering pipeline is simplified: the 3-layer pipeline (PreFilter → EventBatcher → Sampler) is replaced with a lightweight trigger filter and a simple fingerprint-based cooldown. When a trigger passes, the LLM receives the event context plus Grafana MCP tools, and conducts its own investigation.

### Key Decisions

1. **Grafana MCP replaces prometheus-api-client** — Instead of the code making fixed PromQL queries, the LLM gets Grafana MCP as a tool and decides which queries to run. This eliminates blind spots where the hardcoded queries miss the relevant data.
2. **OTLP events are triggers, not data** — Events from the OTLP receiver are used to detect that something may need attention. The actual analysis data comes from querying Prometheus and Loki via MCP, not from the event payload itself.
3. **LLM-driven query autonomy** — The LLM decides which PromQL and LogQL queries to execute. It can start broad ("show me CPU and memory for this pod") and drill down ("show me all error logs from this namespace in the last 10 minutes") based on what it finds.
4. **Prometheus + Loki in v1** — First version supports PromQL (metrics) and LogQL (logs) via Grafana MCP. Tempo (traces) deferred to a future version.
5. **Simplify pipeline to trigger + cooldown** — Remove EventBatcher and Sampler. Replace PreFilter with a simpler trigger filter. Add a lightweight fingerprint-based cooldown to prevent repeated LLM calls for the same issue.
6. **Grafana is an external prerequisite** — Octantis assumes a running Grafana instance with Prometheus and Loki datasources configured. Provisioning Grafana is out of scope.

### Out of Scope

- **Tempo / trace analysis** — Traces are not queried in v1. Can be added later when the MCP integration is proven.
- **Alertmanager integration** — Octantis does not integrate with Grafana alerting rules or Alertmanager. It has its own trigger logic.
- **Multi-tenant / multi-cluster** — Single cluster, single Grafana instance. Multi-tenancy adds complexity not justified at this stage.
- **Dashboard auto-generation** — Octantis does not create or modify Grafana dashboards. It only queries data.
- **K8s API decision** — Whether to keep direct K8s API calls or also route them through MCP is deferred to the Tech Spec.

## 8. Risks

| Risk | Impact | Mitigation |
|------|--------|------------|
| LLM makes too many queries per investigation, increasing cost and latency | High | Query budget (max N queries per investigation) + timeout. Configurable via env var. |
| Grafana MCP latency adds significant time to each investigation | Medium | Set query timeout per MCP call. LLM can decide to proceed with partial data. |
| LLM writes poor PromQL/LogQL queries that return no useful data | Medium | Include example queries in the system prompt. Provide query templates for common patterns (CPU, memory, error rate, pod logs). |
| Removing batching means more LLM calls if trigger filter is too permissive | Medium | Tune trigger thresholds. Fingerprint cooldown prevents repeats. Monitor LLM call rate and adjust. |
| Grafana MCP server unavailability makes Octantis unable to investigate | Medium | Graceful fallback: if MCP is down, LLM analyzes with trigger event data only (degraded mode). Log warning. |
| Breaking change to existing notification flow during refactor | Low | Notification pipeline (plan → notify) is untouched. Only the data-gathering phase changes. |

## 10. Architecture Overview

```
OTel Collector ──OTLP/gRPC:4317──► ┌──────────────────────────────────────────────┐
                                   │  Octantis                                    │
OTel Collector ──OTLP/HTTP:4318──► │  OTLP Receiver                               │
                                   │       ↓                                      │
                                   │  Trigger Filter (simplified PreFilter)        │
                                   │       ↓                                      │
                                   │  Fingerprint Cooldown                         │
                                   │       ↓                                      │
                                   │  LangGraph Workflow                           │
                                   │   ┌─────────────────────────────────────┐    │
                                   │   │ investigate (LLM + Grafana MCP)     │    │
                                   │   │   → PromQL queries (Prometheus)     │◄───┼──► Grafana MCP
                                   │   │   → LogQL queries (Loki)           │    │
                                   │   ├─────────────────────────────────────┤    │
                                   │   │ analyze (LLM: severity from data)  │    │
                                   │   ├─────────────────────────────────────┤    │
                                   │   │ plan (LLM: remediation)            │    │
                                   │   ├─────────────────────────────────────┤    │
                                   │   │ notify (Slack + Discord)           │    │
                                   │   └─────────────────────────────────────┘    │
                                   └──────────────────────────────────────────────┘
```

- **OTLP Receiver** (existing): Receives telemetry events via gRPC/HTTP. Unchanged from PRD 001.
- **Trigger Filter** (new): Simplified version of PreFilter. Detects anomalous metrics, error logs, and critical patterns. Drops noise.
- **Fingerprint Cooldown** (new): Prevents repeated investigations for the same issue within a configurable window.
- **LangGraph Workflow — investigate** (new): LLM receives trigger context + Grafana MCP tools. Autonomously queries Prometheus (PromQL) and Loki (LogQL) to gather evidence.
- **LangGraph Workflow — analyze/plan/notify** (existing): Severity classification, remediation planning, and notifications. Unchanged except that the input data is now richer (queried, not pre-fetched).
