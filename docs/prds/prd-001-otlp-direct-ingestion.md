## 1. Context

- **Product / System**: Octantis — an intelligent infrastructure monitoring agent for EKS/Kubernetes.

- **Current State**: Octantis consumes telemetry events from a Redpanda topic (Kafka-compatible). The current data flow is: OTel Collector → Redpanda → Octantis (aiokafka consumer). Redpanda acts as an intermediary between telemetry exporters and the agent. The project is not yet in production — it exists only in a test environment, developed by a single person.

- **Problem**: Redpanda adds operational and infrastructure complexity without proportional value. Octantis is the sole consumer and final destination of the telemetry data in this pipeline. Running a message broker for a single consumer is unnecessary overhead — it increases the operational surface (provisioning, monitoring, updating, SASL configuration, topic and consumer group management) with no real architectural benefit.

## 3. Proposed Solution

### Overview

Remove Redpanda/Kafka from the ingestion stack and have Octantis expose a native OTLP endpoint (gRPC on port 4317 and HTTP on port 4318). The OTel Collector will be configured to export directly to Octantis, eliminating the intermediary. The rest of the pipeline (PreFilter → EventBatcher → Sampler → LangGraph Workflow) remains unchanged.

### Key Decisions

1. **Support both gRPC and HTTP** — OTLP defines two transports. Supporting both maximizes compatibility with different OTel Collectors and SDKs.
2. **Native OTLP parser** — The current parser in `consumers/redpanda.py:_parse_otel_message()` handles informal JSON. The new parser must consume Protobuf (gRPC) and JSON (HTTP) following the OTLP spec, not the current ad-hoc format.
3. **Complete Redpanda removal** — Clean cut with no transition period. The project is not in production and has no consumers to migrate.
4. **Event loss is acceptable** — If Octantis is down, events are discarded. This is acceptable because if the infrastructure has problems, new events will trigger analysis once the agent comes back.
5. **Parser output unchanged** — The parser continues producing `InfraEvent` (Pydantic model), preserving the interface with the filtering pipeline.

### Out of Scope

- **Retry queue / persistent buffer** — No local persistence layer for events received while Octantis is down. Loss is accepted by design.
- **Event forwarding to other consumers** — Octantis is the final destination. It does not act as a distributor.
- **Changes to the filtering pipeline (PreFilter, Batcher, Sampler)** — The input interface (producing `InfraEvent`) remains the same, so these layers need no changes.
- **Changes to the LangGraph Workflow (collect → analyze → plan → notify)** — Unchanged.
- **Trace processing** — Octantis only processes metrics and logs. The OTLP endpoints accept trace exports for transport compatibility (to avoid breaking OTel Collector pipelines that export all signal types), but trace data is acknowledged and discarded — not parsed or enqueued.
- **Authentication/mTLS on the OTLP endpoint** — For the first version, the endpoint will be internal without authentication. Can be added later if needed.

| F01 | OTLP gRPC Receiver | As an OTel Collector, I want to export telemetry via gRPC to Octantis on port 4317 | 1. Must implement `MetricsService` and `LogsService` from the OTLP gRPC spec 2. | F02 | OTLP HTTP Receiver | As an OTel Collector, I want to export telemetry via HTTP POST to Octantis on port 4318 | 1. Must accept POST on `/v1/metrics` and `/v1/logs` 2. Must accept POST on `/v1/traces` for compatibility — returns HTTP 200 but does not parse or enqueue trace data 3. Supported content-types: `application/json` and `application/x-protobuf` 4. Must return HTTP 200 with empty body on success 5. Port configurable via `OTLP_HTTP_PORT` (default 4318) 6. Disableable via `OTLP_HTTP_ENABLED` (default true) | 1. POST to unknown path → HTTP 404 2. Unsupported content-type → HTTP 415 3. POST to `/v1/traces` → HTTP 200, event ignored |

Payload with resource but no metrics or logs → InfraEvent with metrics=[] and logs=[] 2. Unknown resource attribute → stored in `OTelResource.extra` |
| F04 | Environment variable configuration | As an operator, I want to configure OTLP endpoints via environment variables | 1. `OTLP_GRPC_PORT` (default 4317) 2. `OTLP_HTTP_PORT` (default 4318) 3. `OTLP_GRPC_ENABLED` (default true) — enables/disables the gRPC transport 4. `OTLP_HTTP_ENABLED` (default true) — enables/disables the HTTP transport 5. `OTLP_QUEUE_MAX_SIZE` (default 1000) — max events buffered before drop | 1. Port already in use → clear error message on startup 2. Both transports disabled → warning log, Octantis starts but receives no events |

## 10. Architecture Overview

```
OTel Collector ──OTLP/gRPC:4317──► ┌──────────────────────────────────┐
                                   │  Octantis                        │
OTel Collector ──OTLP/http:4318──► │  OTLP Receiver (new)             │
                                   │       ↓                          │
                                   │  OTLP → InfraEvent Parser (new)  │
                                   │       ↓                          │
                                   │  PreFilter → Batcher → Sampler   │
                                   │       ↓                          │
                                   │  LangGraph Workflow              │
                                   │  (collect → analyze → plan →     │
                                   │   notify)                        │
                                   │       ↓                          │
                                   │  Slack + Discord                 │
                                   └──────────────────────────────────┘
```

- **OTLP Receiver** (new): gRPC + HTTP server that receives telemetry in standard OTLP format. Replaces `RedpandaConsumer`.
- **OTLP → InfraEvent Parser** (new): Converts OTLP Protobuf/JSON payloads into `InfraEvent` (existing Pydantic model). Replaces `_parse_otel_message()`.
- **Filtering Pipeline** (existing): PreFilter, EventBatcher, and Sampler — no changes.
- **LangGraph Workflow** (existing): collect, analyze, plan, notify — no changes.

| Date | Decision | Rationale |
|------|----------|-----------|
| 2026-04-05 | Remove Redpanda/Kafka completely | Octantis is the sole consumer and final destination. Running a broker for a single consumer is complexity without benefit. |
| 2026-04-05 | Support both gRPC and HTTP | Maximizes compatibility with OTel Collectors and SDKs. Some environments use gRPC, others HTTP. |
| 2026-04-05 | Accept event loss if Octantis is down | If the infrastructure has problems, new events will arrive when the agent recovers. Not worth adding buffering complexity for the current use case. |
| 2026-04-05 | Clean cut with no transition | Project is not in production, no users to migrate. |
| 2026-04-06 | Traces accepted but not processed | Octantis only analyzes metrics and logs. Accepting traces at the transport level avoids breaking OTel Collector pipelines that export all signals, but trace data is discarded after acknowledgment. |
| 2026-04-06 | Per-transport enable flags instead of global `OTLP_ENABLED` | `OTLP_GRPC_ENABLED` and `OTLP_HTTP_ENABLED` give operators fine-grained control over which transports are active. |
